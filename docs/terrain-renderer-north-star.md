# North Star: Reverse-Painter Screen-Space Terrain Resolve

This document defines the intended production method for terrain rendering in this repo.

It is intentionally method-first. It does not start from file layout or engine APIs. It starts from the visual contract of the tileset, the current painter-style composition rules, and the cheapest reliable way to answer one question:

> For one screen pixel, which terrain texel owns it, and what are that texel's `RGB`, `height`, `normal`, and `biome`?

The source background for this document is:

- [docs/mathematics-overview.md](./mathematics-overview.md)

## 1. Problem Statement

The renderer must draw a 2.5D isometric sloped terrain built from:

- one packed terrain map in a shader-friendly `8 x packedWidth x packedHeight` representation
- one color tileset atlas per biome
- one packed global surface texture derived from the canonical tileable heightmap
- the fixed 19-slope contract generated by `scripts/tileset.ts`

The map may have arbitrarily many absolute levels in authoring space. The runtime packing codec is allowed to fold those levels into the 8-slice packed representation as long as final screen placement is preserved.

The renderer only cares about:

- `RGB`
- `height`
- `normal`
- `biome`

It does not care about:

- lighting
- shadows
- gameplay physics
- simulation-space collision

## 2. The Correct Mental Model

The terrain renderer is not primarily a geometry problem. It is primarily a composition problem.

Classic isometric tile rendering is traditionally a painter-ordered process:

- tiles are placed in a fixed order
- later tiles visually win where they overlap earlier tiles
- the tile art itself encodes silhouettes, faces, and which lower parts remain visible

This repo's slope tileset was authored for exactly that kind of composition. The current deterministic CPU pipeline goes even further and clips every frame to deterministic ownership masks, but that stronger property is a current asset guarantee, not the long-term mental model we want the renderer to depend on.

The north star is therefore:

- the shader replays the painter logic in reverse for one screen pixel
- it does not reconstruct terrain visibility by solving 3D depth analytically
- it does not use a screen-projected heightmap indirection texture
- it does not raymarch or binary-search the terrain stack

This is a reverse-painter lookup.

## 3. Two Layers of Truth

The renderer should explicitly distinguish between two truths:

### 3.1 Core renderer model

The core renderer model is:

- terrain visibility is defined by painter-style composition order
- for one screen pixel, the first owned hit in reverse painter order is the visible terrain owner

This model remains valid even if future art intentionally reintroduces overlap, such as:

- dangling roots
- hanging cliff details
- decorative lower silhouettes

### 3.2 Current stronger asset property

The current deterministic generator and validator provide a stronger present-day property:

- composed terrain coverage is exact-one under the validated contract
- holes, double ownership, and wrong ownership all fail validation

That property is extremely valuable and should stay documented, but the renderer should not assume that future terrain art will always remain overlap-free.

## 4. Runtime Data Model

The runtime should use three data families.

### 4.1 Packed terrain stack

The authoritative terrain input to the shader is a packed 8-slice stack:

- dimensions: `packedWidth x packedHeight x 8`
- storage: integer array texture
- access mode: nearest, explicit integer fetch, never filtered

Industry-standard target:

- a `texture_2d_array<u32>` style resource
- in Three/WebGPU terms, an integer `DataArrayTexture`-style input is the right shape when tooling allows it cleanly

Each texel is one `PackedTerrainWord`.

At minimum it must contain:

- `shapeRef`
  `0 = empty`, `1..19 = slope frame`
- `biomeIndex`
- reserved bits for future material or variation flags

The renderer treats the exact packed `x/y` remap and level compensation as codec internals. Those details belong to the packing codec, not to the core renderer idea.

### 4.2 Biome color atlas array

The terrain color source is one atlas per biome, normalized into a texture array when possible.

Requirements:

- same frame order across all biome layers
- same frame size
- same spacing and margins
- same placement contract
- same ownership semantics in alpha

The color atlas remains the canonical source of `RGB`.

### 4.3 Global terrain surface texture

The renderer also carries one packed surface texture for the whole map, derived from the canonical tileable heightmap with the shared `tileableHeightmapToTileData -> tileDataToTerrain -> packTerrain` path.

The surface texture exists so the renderer can read the ground point's:

- `height`
- `normal`

with one ground lookup after color ownership is resolved.

Recommended packing:

- `RGB`: world-space normal
- `A`: normalized global surface height

The winning pixel's `biome` still comes from the winning `PackedTerrainWord`.

## 5. Per-Pixel Resolve Algorithm

The production method has two layers:

- the base renderer contract, which must work for any compatible future atlas family
- the current-atlas deterministic specialization, which is allowed to exploit stronger present-day guarantees

### 5.1 Base renderer contract

The base resolve method is:

1. Start from the screen pixel currently being shaded.
2. Derive the packed-map lookup origin for that pixel.
3. Enumerate the bounded set of terrain candidates in exact reverse painter order.
4. For each candidate:
   - fetch the `PackedTerrainWord`
   - skip if `shapeRef == 0`
   - decode `shapeRef` and `biomeIndex`
   - compute the atlas UV for this pixel against that candidate, including the codec-owned level and slice compensation
   - sample color atlas alpha
   - if alpha says the candidate owns this pixel, stop immediately
5. Sample the winning candidate's color atlas once for `RGB`
6. Sample the packed global surface texture once at the resolved ground point for `height` and `normal`
7. Return the resolved terrain payload

### 5.2 Current-atlas specialization

For today's deterministic atlas family, the fast path should not be “run the same loop and hope early exit helps.”

It should be a real specialization:

- skip candidates that can never own the current residue
- replace per-candidate atlas-alpha sampling with precomputed integer bitset ownership tests
- still preserve exact reverse painter order and first-owned-hit semantics

In other words, the fast path changes how ownership is tested, not what visibility means.

The returned payload is:

```ts
type TerrainResolveResult = {
  color: [number, number, number];
  height: number;
  normal: [number, number, number];
  biome: number;
  shapeRef: number;
};
```

## 6. What Does One Screen Pixel Look Up?

For one terrain pixel on screen in the base path:

- `RGB` comes from the winning biome color atlas texel
- ownership comes from that atlas texel's alpha
- `height` comes from one global surface fetch at that pixel's resolved ground point
- `normal` comes from one global surface fetch at that pixel's resolved ground point
- `biome` comes from the winning packed terrain word

So the base cost model is:

- one packed-map fetch per candidate
- one atlas alpha fetch per candidate
- one final color fetch for the winner
- one final global surface fetch for the ground point

There is no screen-projected terrain lookup texture in the base method.

There is no geometric depth reconstruction in the base method.

There is no per-pixel search for the nearest 3D triangle in the base method.

For the current-atlas deterministic specialization:

- ownership does not come from candidate atlas-alpha reads
- ownership comes from a precomputed bitset derived from the same deterministic ownership oracle that produced the current atlas
- once a winner is found, the renderer still performs the usual one final color fetch and one final global surface fetch

## 7. Painter Traversal Reflection

The document should make the painter reasoning explicit because this is the heart of the method.

### 7.1 Current classic order in the repo

The repo currently composes terrain according to the same placement contract used by:

- `src/game/lib/tilemap.ts`
- `scripts/validate-terrain-coverage.ts`

The important current facts are:

- Tiled map render order is `right-down`
- layers are emitted as `level-0`, `level-1`, `level-2`, ...
- placement walks layer order first, then row order, then column order
- tile screen placement uses the fixed native `128 x 96 / 64 / 16` contract

That means the CPU side already has a canonical painter composition order for the current atlas system.

### 7.2 Reverse traversal

The shader must traverse the candidate set in the exact reverse of that painter order.

This is the core requirement:

- if the CPU painter would draw tile `A`, then tile `B`
- and `B` visually wins at a screen pixel
- then the shader must encounter `B` before `A` during reverse traversal

The first owned hit is therefore the visible result.

### 7.3 Why this matters more than “no overlap”

The stronger current deterministic contract means:

- today's atlas system usually yields exact-one ownership

But the north-star renderer must still be based on reverse painter order because:

- painter order is the stable industry-standard composition model
- it remains correct if future atlases add deliberate overlap
- it describes the actual artistic control surface

Exact-one ownership is a present-day strength check, not the main abstraction.

## 8. Candidate Bound

The candidate count is now considered proven for the current screen contract.

### 8.1 Placement equations

For a packed candidate, screen placement follows:

```text
left = 64 * d
top  = 32 * s - 32 - 16 * slice
```

where:

- `d = packedX - packedY`
- `s = packedX + packedY`
- `slice in [0, 7]`

Define:

```text
q = 2s - slice
top = 16q - 32
```

For a fixed screen pixel, the local frame coordinates inside a candidate are:

```text
u = screenX - 64d
v = screenY + 32 - 16q
```

### 8.2 Hard geometric window

The native frame contract is:

- frame width `128`
- frame height `96`
- horizontal stripe spacing `64`
- vertical stack step `16`
- packed slices `8`

From those equations:

- the horizontal overlap condition admits exactly two diagonal `d` stripes for any pixel
- for a fixed stripe and slice, the vertical overlap condition admits three consecutive `s` rows before parity filtering
- parity between `s` and `d` reduces that to at most one or two rows per slice

This yields the hard coarse decomposition:

```text
2 stripes x 6 vertical bands x 2 stack entries = 24 raw candidates
```

Equivalently:

- each stripe has a coarse cap of `12`
- the full base renderer has a hard cap of `24 = 12 + 12`

This is the compatibility bound for the renderer. It comes from placement math alone, not from any special property of today's atlas art.

### 8.3 Band structure

The `q` formulation reveals that the 24 raw candidates are not one undifferentiated list.

They are organized as:

- 2 diagonal stripe streams
- 6 vertical bands per stripe
- 2 stack entries per band

The two stack entries in one band are the pair:

```text
(s, slice) and (s + 2, slice + 4)
```

They land on:

- the same screen footprint
- the same local `(u, v)` inside the frame

For the current atlas family, that means a band can share one ownership test for both stack entries.

### 8.4 Current-atlas bound

Brute-force evaluation over the fundamental residue cell for the current 19-shape deterministic atlas family gives a stronger present-day result:

- at most `16` packed positions can ever be possible visible winners for one screen pixel

Because the coarse structure is 2-deep per band, that immediately implies:

- at most `8` live bands for any pixel in the current atlas family

This `16` / `8` result is a current-atlas specialization, not the long-term renderer contract.

The renderer therefore has two valid bounds:

- base compatibility bound: `24`
- current-atlas fast-path bound: `16` possible winners, or equivalently `8` live bands

## 9. PackedTerrainCodec Boundary

The renderer should not expose the exact packed `x/y` arithmetic as its main idea.

That logic belongs to `PackedTerrainCodec`.

The codec owns:

- folding arbitrary absolute levels into the 8-slice packed representation
- any packed-space `x/y` translation needed by that fold
- any level compensation term needed to evaluate atlas ownership for a candidate
- the reverse mapping used for debugging and CPU reference validation

The renderer only needs these outputs:

- packed terrain stack
- packed lookup origin for a screen pixel
- exact reverse painter candidate iterator
- or, for the optimized shape, two reverse-painter stripe streams plus a merged traversal helper
- atlas-space compensation inputs for a candidate

This keeps the shader method stable even if the packing arithmetic changes later.

The current-atlas specialization must respect this boundary:

- the codec still owns packed folding
- the codec still owns exact reverse painter order
- the fast path only contributes a static ownership filter derived from the validated atlas family

The fast path is therefore an ownership acceleration layer, not a second visibility model.

## 10. Deterministic Specialization

Today's production atlas family gives three unusually strong facts:

1. the placement contract is fixed at `128 x 96 / 64 / 16 / 8`
2. the current atlas replaces frame alpha with deterministic ownership masks from [scripts/lib/terrain-ownership.ts](../scripts/lib/terrain-ownership.ts)
3. [scripts/validate-terrain-coverage.ts](../scripts/validate-terrain-coverage.ts) enforces exact-one native coverage and guarantees owned pixels do not expose undefined RGB

That means the current-atlas fast path should do two things:

- skip slots or bands that can never own the current residue
- replace per-candidate atlas-alpha sampling with tiny integer bitset tests

It should not keep doing one atlas-alpha fetch per candidate in deterministic-atlas mode. That would miss the biggest specialization win the current atlas family makes available.

### 10.1 Phase 1: slot-oriented LUT

The first implementation should keep the codec boundary exactly as it is and precompute:

```ts
type NativeExactSlotLUT = {
  slotCount: Uint8Array; // [64][32]
  activeSlotList: Uint8Array; // [64][32][16]
  ownerShapeSet: Uint32Array; // [64][32][24], 19 bits used
  revision: NativeExactRevision;
};
```

Meaning:

- for each residue `rx in [0, 63]`, `ry in [0, 31]`
- ask the CPU reference codec for the 24 contract slots in exact reverse painter order
- compute which `shapeRef` values can own that slot at that residue
- store that as a 19-bit mask
- keep only the nonzero slots in `activeSlotList`
- assert `slotCount <= 16`

Runtime shape:

```ts
resolveNativeExactFast(screenX, screenY):
  rx = floorMod(screenX, 64)
  ry = floorMod(screenY, 32)
  count = slotCount[rx][ry]

  for i in 0 .. count - 1:
    slotOrdinal = activeSlotList[rx][ry][i]
    candidate = codec.getCandidateByOrdinal(screenX, screenY, slotOrdinal)
    word = fetchPackedTerrainWord(candidate)
    shapeRef = decodeShapeRef(word)
    if shapeRef == 0:
      continue
    if bit(ownerShapeSet[rx][ry][slotOrdinal], shapeRef) == 0:
      continue
    return sampleWinnerColorAndMetadata(candidate, word)
```

This version is the easiest one to prove and the easiest one to wire into the existing codec contract.

Its memory footprint is still small enough to be practical, on the order of roughly `200 KB` total for the masks plus compact slot lists.

### 10.2 Phase 2: band-oriented LUT

The preferred end state should compress the specialization to the actual band structure:

```ts
type NativeExactBandLUT = {
  liveBandCount: Uint8Array; // [64][16]
  liveBandList: Uint8Array; // [64][16][8]
  bandShapeSet: Uint32Array; // [64][16][12], 19 bits used
  revision: NativeExactRevision;
};
```

with:

```text
bandId = stripe * 6 + band
stripe in {0, 1}
band in {0..5}
```

For residue:

```text
rx = floorMod(screenX, 64)
ry = floorMod(screenY, 16)
```

the local frame coordinates are:

```text
right stripe: u = rx
left stripe:  u = rx + 64
band b:       v = ry + 16 * b
```

Runtime shape:

```ts
resolveNativeExactFast(screenX, screenY):
  rx = floorMod(screenX, 64)
  ry = floorMod(screenY, 16)
  count = liveBandCount[rx][ry]
  stripes = codec.getTwoReversePainterStripes(screenX, screenY)

  for bandRef in codec.mergeLiveBands(stripes, liveBandList[rx][ry], count):
    shapeSet = bandShapeSet[rx][ry][bandRef.bandId]

    if bandRef.hasFront:
      word0 = fetchPackedTerrainWord(bandRef.front)
      shape0 = decodeShapeRef(word0)
      if shape0 != 0 && bit(shapeSet, shape0) == 1:
        return sampleWinnerColorAndMetadata(bandRef.front, word0)

    if bandRef.hasBack:
      word1 = fetchPackedTerrainWord(bandRef.back)
      shape1 = decodeShapeRef(word1)
      if shape1 != 0 && bit(shapeSet, shape1) == 1:
        return sampleWinnerColorAndMetadata(bandRef.back, word1)
```

This is the version that matches the actual search structure best:

- up to 8 live bands
- 2 packed entries per live band
- 0 candidate atlas-alpha fetches

Its memory footprint is substantially smaller than the slot-oriented version, on the order of roughly `57 KB` total.

### 10.3 Enablement and fallback

The specialization is valid only when all of these are true:

- the runtime path is using the current deterministic atlas contract
- the lookup tables were generated from the same 19-frame order, scene spec, ownership masks, and `128 x 96 / 64 / 16 / 8` placement contract
- the runtime path is using the same native screen contract assumed by the validator
- the stored fast-path revision matches the active atlas family

Otherwise the renderer must fall back to the base `24`-candidate reverse-painter path.

## 11. Why This Is Not a Geometry-Depth Renderer

This north star explicitly rejects a different class of solutions:

- reconstructing the frontmost fragment by comparing analytic local depth between multiple candidates
- solving visibility through 3D triangle depth math in the fragment shader
- binary searching for the terrain layer behind a screen pixel
- rendering instanced quads and relying on fragment depth as the primary ownership truth

Those approaches are valid in other engines, but they are not the best fit for this tileset contract.

The atlas already encodes the visual answer.

The shader should ask:

- “who would have been painted last here?”

not:

- “which 3D surface is mathematically nearest?”

## 13. Validation Requirements

The north-star design is only valid if all of these are proven.

### 13.1 Painter traversal proof

- derive the reverse traversal from the repo's actual placement contract
- prove it matches current `right-down` Tiled composition plus level ordering
- build a CPU reference compositor that uses the same traversal

### 13.2 Candidate bound proof

- derive the exact `24 = 12 + 12` hard bound from the placement equations
- prove the `2 stripes x 6 bands x 2 stack entries` decomposition explicitly
- validate the bound with brute-force residue sweeps
- fail if any fixture requires a candidate outside the derived window

### 13.3 Current-atlas fast-path proof

- document the current-atlas observation that the present deterministic slope family reduces possible winners to `16`
- derive the corollary that the present atlas family therefore has at most `8` live bands
- generate fast-path tables from the same ownership oracle trusted by the validator
- compare fast-path winners against the CPU reference compositor in exact reverse painter order
- store a revision hash beside the generated tables and disable the specialization if it no longer matches the active atlas family

### 13.4 Current-generator parity

- reproduce the current deterministic visible winners exactly
- verify parity against the CPU reference compositor
- keep the existing no-hole / no-overlap / wrong-owner checks in the loop

### 13.5 Future-overlap safety

- introduce synthetic atlases with deliberate lower-part overlap
- verify that reverse painter resolve still chooses the intended winner
- do not assume exact-one ownership forever

### 13.6 Performance checks

- base path worst case:
  - up to `24` packed-map fetches
  - up to `24` atlas-alpha fetches
  - `1` final color fetch
  - `1` final metadata fetch
- minimal current-atlas fast path worst case:
  - up to `16` packed-map fetches
  - up to `16` bitset tests
  - `0` candidate atlas-alpha fetches
  - `1` final color fetch
  - `1` final metadata fetch
- recommended current-atlas fast path worst case:
  - up to `8` live-band lookups
  - up to `16` packed-map fetches
  - `0` candidate atlas-alpha fetches
  - `1` final color fetch
  - `1` final metadata fetch
- no screen-projected indirection texture
- no geometry-depth reconstruction in the base renderer

### 13.7 Shipping order

- ship the slot-oriented LUT first because it is the easiest version to prove and the easiest one to wire into the existing codec boundary
- treat the band-oriented LUT as the intended follow-up once the traversal API is explicitly stripes and bands

## 14. Non-Goals

This document is not trying to define:

- terrain lighting
- terrain shadowing
- world physics
- gameplay collision
- foliage, props, or units
- the final file/module split of the implementation

It only defines the terrain resolve method.

## 15. Final Position

The north star for this repo is:

- a packed terrain map
- biome atlas arrays
- a reverse-painter screen-space resolve shader
- atlas alpha deciding ownership in the base path
- a deterministic-atlas-only ownership-bitset specialization for the current atlas family
- one final metadata fetch giving height and normal in both paths

The current deterministic pipeline is stronger than that and should remain celebrated and validated, but it should be treated as today's high-quality asset guarantee, not as the only possible long-term terrain art contract.
